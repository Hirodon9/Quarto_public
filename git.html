はじめてのGit / GitHub

本稿は，GitおよびGitHub初心者が，単独でこの文書のみを参照しながら作業を進められるように構成された手引きである．手順の提示にとどまらず，各操作がGitの内部構造上何を意味するかについても，必要な箇所で言及する．用語数は多いが，これらを暗記する必要はなく，作業の流れの中で順次理解が形成されることを想定している．

認証にはSSHキーを用いる方式を採用する．設定はGitHubへの接続を行う章でまとめて行う．

最終的な到達点は，自分のWebページをGitHub上に公開することである．所要時間の目安は80〜110分．

このガイドの地図

本稿では以下の6段階を扱う．現在の位置を失った場合は，この地図に立ち返ること．

  1. 事前準備（ターミナルおよび作業フォルダの用意．この段階ではGitHubは扱わない）
  2. Gitの仕組みについて，作業に先立って把握する
  3. 自分のPCの中だけで，ファイルの変更履歴を記録する作業（Git）
  4. GitHubのアカウントを作成し，SSHキーによって認証を設定した上で，記録をネット上の保管庫に送信する
  5. コマンドによって行った操作を，マウス操作でも可能なツール（VS Code）上で確認する
  6. 作成したファイルをWebページとして公開する（GitHub Pages）

Gitは，Wordの「変更履歴」機能や，ゲームにおける「セーブポイント」に近い性質を持つ道具であると言える．ファイルに対して「ある時点までの状態」を記録しておき，後からその状態を参照・復元できる，という機能を提供する．以下の内容を完全に理解できなくても支障はなく，作業を進める中で理解が形成される．

0. 事前準備

本節では，GitHubはまだ扱わない．手元のPC上で，以降の作業に用いる環境を整える．

0-1. ターミナルを開く

Cmd + Spaceを押して検索バーを表示し，「ターミナル」と入力してEnterキーを押す．画面が開けば作業環境の準備は完了である．以降，コマンドの入力はこの画面で行う．

0-2. Gitの導入状況を確認する

ターミナル上で以下を実行する．以降，コードブロックはそのままターミナルに貼り付けて実行するものとする．

git --version

git version 2.xx.xのような表示が得られれば，Gitは既に利用可能な状態にある．「Command Line Toolsをインストールしますか」という表示が出た場合は「インストール」を選択し，完了後に同一のコマンドを再実行する．

0-3. ターミナルの基本操作（3種）

以降，フォルダの移動を行う操作が頻出するため，先に以下の3種を確認する．

コマンド 機能
pwd 現在の位置を表示する
ls 現在の位置にあるファイルの一覧を表示する
cd フォルダ名 指定したフォルダへ移動する

これは，Finder上でフォルダを選択して移動する操作を，文字列による指示として行っているものに過ぎない．

Note

本稿で扱うファイル編集は，echoコマンド（文字列をファイルへ書き込む単純な操作）と，後述するVS Codeのエディタのみで完結する．vimのようなターミナル上の高機能エディタは，本稿の範囲では一切使用しない．初学者にとって導入の障壁が高いため，意図的に避けている．

0-4. 作業用フォルダの作成

デスクトップやドキュメントフォルダの内部では，macOSによるターミナルアクセスの保護機能により，原因の特定しにくいエラーが生じることがある．そのため，ホームフォルダの直下に作業フォルダを作成する．

cd ~
mkdir git-practice
cd git-practice
pwd

最後のpwdの実行結果として，/Users/ユーザー名/git-practiceのような表示が得られれば，作業位置は正しい．

Important

以降，git initというフォルダの初期化を行うコマンドを用いる．その実行直前には，必ずpwdによって「意図したフォルダ内に位置しているか」を確認する習慣をつけること．これを怠ると，想定より広い範囲（最悪の場合はホームフォルダ全体）を初期化してしまう可能性がある．なお，誤って実行した場合も，当該フォルダでrm -rf .gitを実行すれば取り消すことができる（フォルダ内の他のファイルは削除されない）．

Warning

このあたりで詰まったら

  • gitまたはpythonに対して「command not found」と表示される場合，0-2のインストールが完了していない可能性がある．ターミナルを再起動し，git --versionを再実行する．
  • cdの実行時に「そのようなファイルやディレクトリはありません」と表示される場合，フォルダ名の入力誤り，またはmkdirが未実行である可能性がある．lsによって現在位置にあるものを確認する．

1. 作業に先立って：Gitの仕組みを把握する

コマンドの実行に先立ち，Gitが内部でどのような「場所」を扱っているかを把握しておくことで，以降のaddやcommitという操作が，意味の不明な手続きとしてではなく，対象が明確な操作として理解できるようになる．

1-1. 3つの場所

Gitを利用している際，フォルダの内部には，目に見えない形で以下の3つの場所が存在している．

ワーキングツリー  →  インデックス（ステージングエリア）  →  リポジトリ
（現在編集中のファイル）  （次の記録対象の仮登録場所）       （記録そのものの保管庫）

Definition 1 (ワーキングツリー)
ワーキングツリーとは，現在エディタ上で編集している，通常のファイルそのものを指す．

Definition 2 (インデックス（ステージングエリア）)
インデックス（ステージングエリア）とは，「次の記録にこのファイルを含める」という仮の決定を保持しておく場所を指す．Web開発の文脈において，本番公開に先立って動作を試験する環境をステージング環境と呼ぶことがあるが，これと類似した位置づけにあると言える．本番（記録）への登録に先立つ，一段階手前の控え室にあたる．

Definition 3 (リポジトリ)
リポジトリとは，記録（履歴）が積み重なっていく保管庫を指す．その実体は，フォルダ内部に生成される.gitという名称の隠しフォルダである．

git addは，Definition 1からDefinition 2への仮登録を行う操作であり，git commitは，Definition 2にある内容をDefinition 3へ正式に記録する操作である．この2段階に分離されている理由は，「編集中のファイルが10個あっても，記録の対象としたいのはそのうち3個のみである」という場合のように，記録の対象を選別した上で記録を行うことを可能にするためである．

1-2. コミットは履歴上の1単位である

Gitの履歴は，「丸とそれを結ぶ線」という図によって説明されることが多い．

○ ─ ○ ─ ○ ─ ○   ← 丸の各々が1つの「コミット」，すなわちある時点の状態の記録（スナップショット）

Definition 4 (コミット)
コミットとは，ある時点におけるファイル全体の状態を，まとめて記録したものを指す（名詞としての用法）．また，記録を行う動作そのもの（履歴上の丸を1つ追加する操作）も同一の語で呼ばれる（動詞としての用法）．丸と丸を結ぶ線は，隣接するコミット間の差分（変更内容）を表すものと解釈できる．

ゲームにおけるセーブポイントに例えれば，丸の各々が「セーブデータ」に相当する．3日前の状態へ復帰したい場合，3日前の丸の位置へ切り替えればよい，という発想である．なお，過去の状態への実際の切り替え操作については，本稿の範囲では扱わない．現段階では，「復帰可能な記録が積み重なっていく」という理解で十分である．

1-3. 用語の整理

Definition 3，Definition 1に関連して，以下の用語を併せて整理する．

Definition 5 (ローカルリポジトリ／リモートリポジトリ)
ローカルリポジトリとは，自分のPC内部に存在するDefinition 3を指す．リモートリポジトリとは，サーバー（本稿の場合はGitHub）の内部に存在するDefinition 3を指す．

用語 意味
プロジェクト 管理対象とするファイル・フォルダ一式（卒業論文であれば，本文・図・参考文献等の一括）
リポジトリ Definition 3を参照
ワーキングツリー Definition 1を参照
ローカルリポジトリ／リモートリポジトリ Definition 5を参照

後述するgit pushは，「ローカルリポジトリの記録を，リモートリポジトリ（GitHub上の保管庫）へ送信する」操作であると言い換えられる．現段階ではローカルリポジトリのみが存在するため，次節ではその範囲に限定して扱う．

2. ローカルでGitを使用する最小サイクル

ここがGitの核心部分にあたる．init → add → commitの3操作を扱う．Section 1.3で述べた3つの場所が，実際にどのように機能するかを，本節で確認する．GitHubはまだ扱わない．対象は自分のPC内部に限定される．

2-1. 練習用ファイルの作成

echo "はじめてのファイル" > memo.txt
ls

memo.txtというファイルが生成されていれば成功である．これは現時点ではDefinition 1にのみ存在するファイルであり，Gitの記録には一切登場していない．

2-2. フォルダをGitの管理対象とする

git init

これにより，このフォルダ内部に.gitという隠しフォルダが生成され，これがDefinition 3となる．この時点では記録は何も行われておらず，「記録のための箱」が設置された状態にある．

2-3. 名前およびメールアドレスの設定（最初の1回のみ）

git config --global user.name "氏名"
git config --global user.email "メールアドレス@example.com"

これは「誰が変更を行ったか」を記録に残すためのラベルである．第3節でGitHubのアカウントを作成した後，そこに登録するメールアドレスと一致させておくと，後の混乱を避けられる．

2-4. インデックスへの登録（add）

git add memo.txt
git status

git statusの実行結果として，現在の状態が表示される．memo.txtが「記録対象として登録される」旨の表示（緑色で表示される）になっていれば成功である．

これはDefinition 1からDefinition 2への仮登録に相当する．addの時点では，Definition 3には何も記録されていない．Definition 2という控え室に置かれた状態にとどまる．

2-5. 記録の実行（commit）

git commit -m "最初のファイルを追加"

これにより，Definition 2にあった内容が，Definition 3へ正式に記録される．履歴の図において，丸（コミット）が1つ生成された状態であり，最初の「セーブポイント」にあたる．-mに続く文字列は，後から参照した際に「これがどのような変更であったか」を識別するための記述である．

2-6. サイクルの再実行

同一の手続きを繰り返すことで，操作が定着する．同様の流れをもう一度実行する．

echo "2行目を追加" >> memo.txt
git status
git add memo.txt
git commit -m "2行目を追加"
git log

git logの実行により，これまでの履歴（丸が2つ連なったもの）が表示される（qキーにより表示から退出できる）．各行が1つのコミット，すなわち1つの丸に対応している．

これがGitの基本サイクルである：ファイルの変更 → add（Definition 2への仮登録） → commit（Definition 3への正式な記録）．この手続きの繰り返しにより，任意の過去の時点に復帰可能な履歴が形成されていく．現時点では，この履歴は自分のPC内部にのみ存在する．次節では，これをGitHubへ送信する．

Warning

このあたりで詰まったら

  • git commitの実行時に「Please tell me who you are」と表示される場合，2-3の設定が未実行である．当該設定を行った上でgit commitを再実行する．

3. GitHubのアカウント作成，SSHキーによる認証，および接続

ここまでは自分のPC内部に限定された話であった．本節では，その記録をネット上の保管庫（GitHub，すなわちリモートリポジトリ）へ送信する．PCの外部へ記録を送信するということは，GitHub側が「送信者が本人であるか」を確認できなければならない．この確認のための仕組みがSSHキーである．まず仕組みを把握し，その後設定を進める．

3-1. 認証が必要となる理由

「権利を有する本人であること」を確認する手続きを認証と呼ぶ．認証には大別して3種が存在する．

Definition 6 (認証の3分類)

  • 所持認証：特定の物を所持していることによって確認する方式（自宅の鍵を所持していること＝入室の権利を有することの根拠となる）
  • 知識認証：特定の知識を有することによって確認する方式（ナンバー錠の番号を知っていること）
  • 生体認証：身体の一部によって確認する方式（指紋・顔認証）

GitHubはネットワークの先方に存在するサーバーであるため，物理的な鍵を提示することはできない．そこで採用されるのが，データ上の「鍵」による所持認証，すなわちSSH公開鍵認証である．

Definition 7 (SSH公開鍵認証)
SSH公開鍵認証とは，以下の手順によって成立する所持認証の一方式である．

  1. ssh-keygenコマンドにより，秘密鍵と公開鍵という，対になった2つのデータを生成する
  2. 公開鍵のみをGitHubへ登録する（公開鍵はその名称の通り，第三者に見られても問題のないデータである）
  3. 秘密鍵は自分のPC内部にのみ保持し，第三者へ渡すことはしない
  4. pushの実行時，GitHubは「対になる秘密鍵を所持しているか」を確認し，一致した場合に本人として認証する

公開鍵は秘密鍵から生成可能であるが，逆（公開鍵から秘密鍵を生成すること）は不可能な構造になっている．したがって，公開鍵のみを第三者に提示しても安全である，という点がこの認証方式の前提となる．

なお，3-3で生成する鍵には暗号アルゴリズムの種別が存在し，ssh-keygenの-tオプションによって指定する．以下に主な種別を示す．

アルゴリズム 位置づけ
ed25519 現在の推奨．鍵長が短い一方で安全性が高く，処理速度も優れる
rsa 古くから用いられてきた標準．現在も利用可能であるが，ed25519が利用可能な環境では選択する必要性は低い

本稿ではed25519を採用する．

なお，秘密鍵自体にかけるパスフレーズ（3-3で設定する）は，もう一段階の鍵に相当する（PCが盗難された場合でも，パスフレーズを知らない者は秘密鍵を利用できない）．macOSのキーチェーンに登録しておけば，日常の操作において繰り返し入力を求められることはない．

3-2. GitHubアカウントの作成

ブラウザでhttps://github.comを開き，右上の「Sign up」よりアカウントを作成する．

Note

ユーザー名は今後継続的に第三者から参照される可能性があるため，過度に簡略な名称は避けることが望ましい．

3-3. SSHキーの生成

ターミナル上で以下を実行する（メールアドレスの部分はGitHubに登録したものに置き換える）．

ssh-keygen -t ed25519 -C "メールアドレス@example.com"

以下の3点が問われるため，それぞれ次のように応答する．

  1. Enter file in which to save the key (...)：何も入力せずEnterキーを押す（保存先は既定のままでよい）
  2. Enter passphrase (empty for no passphrase):：任意のパスフレーズを入力しEnterキーを押す（入力内容は画面に表示されないが，正しく受け付けられている）
  3. Enter same passphrase again:：確認のため同一のパスフレーズを再入力する

これにより，~/.ssh/フォルダ内に2つの鍵（id_ed25519およびid_ed25519.pub）が生成される．これがDefinition 7で述べた秘密鍵・公開鍵の対である．

3-4. 公開鍵のGitHubへの登録

生成された2つのファイルのうち，.pubが付与された側（公開鍵）のみをGitHubへ登録する．以下のコマンドにより，公開鍵の内容をクリップボードへコピーする．

pbcopy < ~/.ssh/id_ed25519.pub

（pbcopyはmacOS専用の，クリップボードへの内容コピーを行うコマンドである．）

続いてブラウザでGitHubを開き，右上の自分のアイコンから「Settings」→左側メニューの「SSH and GPG keys」→「New SSH key」を選択する．

  • Title：任意の名称（例：「自分のMac」）
  • Key type：Authentication Keyのまま変更しない
  • Key：先にコピーした内容を貼り付ける（Cmd + V）

「Add SSH key」を選択すれば登録は完了する．

3-5. 接続試験，およびパスフレーズのmacOSへの登録

ターミナル上で以下を実行する．

ssh-add --apple-use-keychain ~/.ssh/id_ed25519

これは，先に設定したパスフレーズをmacOSのキーチェーン（パスワード管理機能）へ登録する操作である．この際に一度のみパスフレーズの入力が求められる．以降，SSHキーの利用時にパスフレーズが問われることはなくなる．

最後に，GitHubとの接続が正しく確立されるかを試験する．

ssh -T git@github.com

初回実行時には「The authenticity of host ‘github.com’ can’t be established. … Are you sure you want to continue connecting (yes/no)?」という表示が出るため，yesと入力しEnterキーを押す．

続いて以下のような表示が得られれば成功である．

Hi ユーザー名! You've successfully authenticated, but GitHub does not provide shell access.

これにより，以降のpushにおいて認証の段階で停止することはなくなる．

3-6. GitHub上での空のリポジトリの作成

ブラウザでGitHubへログインし，右上の「+」より「New repository」を選択する．

  • Repository name：git-practice
  • Public／Private：いずれでもよいが，初学者には判別しやすいPublicを推奨する
  • 「Add a README file」等のチェックボックスはいずれも選択しない（空の状態で作成する）

作成後，コマンド例が表示される画面に移行する．「HTTPS」「SSH」の切り替えが存在するため，「SSH」側を選択した上で，表示されたコマンドをターミナルに貼り付けて実行する．概ね以下の形式となる（ユーザー名の部分は実際の表示に従う．URLがgit@github.com:...の形式であることを確認する）．

git remote add origin git@github.com:ユーザー名/git-practice.git
git branch -M main
git push -u origin main

remote addは，ローカルリポジトリとリモートリポジトリを接続する宣言であり，pushは，ローカルリポジトリにある記録を，接続先（リモートリポジトリ）へ実際に送信する操作である．先に設定したSSHキーにより，この段階で認証のやり取りに起因する停止は生じない．

3-7. 結果の確認

GitHubのリポジトリページをブラウザで再読み込みする．memo.txtが表示されていれば成功である．ターミナル上で作成した2件のコミットの記録も，同画面より確認できる．すなわち，これまで自分のPC内部にのみ存在していた履歴上の2つの丸が，GitHub上のリモートリポジトリにも同一の形で存在する状態にある．

Warning

このあたりで詰まったら

  • ssh-keygenの実行時に「already exists. Overwrite?」と表示される場合，既に鍵が存在する状態にある．上書きを望まない場合はnと応答し，3-3を省略して3-4へ進めばよい（既存の鍵を用いる）．
  • ssh -T git@github.comの実行時に「Permission denied (publickey)」と表示される場合，3-4の登録が正しく完了していない可能性がある．GitHubの「SSH and GPG keys」のページにおいて，鍵が登録されているかを確認する．
  • pushの実行時に「Permission denied (publickey)」と表示される場合，3-4の公開鍵登録が未完了であるか，3-6において「SSH」側のURL（git@github.com:...の形式）が用いられていない（HTTPSのURLになっている）可能性がある．git remote -vにより現在のURLを確認する．
  • remote addの実行時に「remote origin already exists」と表示される場合，git remote -vにより現状を確認し，必要であればgit remote remove originを実行した上で3-6を再実行する．

4. 既出コマンドの意味の振り返り

マウス操作による方法へ移行する前に，既出の5つのコマンドが何を行うものであったかを，自身の言葉で説明できるかを確認する．

コマンド 行った操作 第1節の用語による言い換え
git init このフォルダを記録対象とする .gitフォルダ（Definition 3）の生成
git add 次の記録に含めるファイルを選択する Definition 1からDefinition 2への仮登録
git commit 選択した内容をセーブポイントとして記録する Definition 2からDefinition 3への記録（丸を1つ追加）
git remote add ローカルとGitHub上の場所を接続する Definition 5の接続（SSH経由）
git push ローカルの記録をGitHubへ送信する ローカルリポジトリからリモートリポジトリへの送信（SSHキーによる認証済み）

以上が説明可能であれば，以降マウスによって行う操作についても，「既出の操作と同一である」という対応関係が把握できるはずである．

5. VS CodeのGUIによる同一操作の実行

コマンドの意味を把握した上で，同一の操作がマウスによっても可能であることを確認する．SSHキーはGit自体の設定（~/.ssh/の内容）に紐づくものであるため，VS Codeにおいても特別な追加設定なしにそのまま利用できる．

5-1. VS Codeの導入（未導入の場合）

https://code.visualstudio.comを開き，ダウンロードした上で導入する．

5-2. 練習フォルダを開く

VS Codeを起動し，「File」→「Open Folder」より，先に作成したgit-practiceフォルダを開く．

5-3. Source Controlパネルの確認

左側に縦に並んだアイコン群のうち，分岐した形状のアイコン（「Source Control」）を選択する．

ターミナル上でgit statusを実行した際に表示された内容（Definition 1とDefinition 2の差分）が，ここに視覚的に表示されている．

5-4. ファイルの変更と，GUIによるcommitおよびpush

まずファイルに変更を加える（VS Codeのエディタでmemo.txtを直接開いて1行追加してもよく，ターミナルで以下を実行してもよい）．

echo "3行目はVS Codeから" >> memo.txt

VS Code上での操作は以下の3手順である．

  1. Source Controlパネルにmemo.txtが「変更あり」として表示される．ファイル名の右側にある「+」ボタンを選択する（これはgit add，すなわちDefinition 2への仮登録に相当する操作である）．
  2. パネル上部のメッセージ入力欄にコミットメッセージ（例：「3行目を追加」）を入力し，上部のチェックマーク（✓）ボタンを選択する（これはgit commit，すなわちDefinition 3への記録に相当する操作である）．
  3. 画面下部または左下に表示される「Sync Changes」または雲の形状のアイコンを選択する（これはgit push，すなわちリモートリポジトリへの送信に相当する操作である）．SSHキーが既に設定済みであるため，ログイン画面等が表示されることなく送信が完了するはずである．

ボタンの名称は異なるが，内部的には先にターミナルで実行したadd・commit・pushと同一の処理が行われている．コマンドの意味が把握されていれば，ボタンの意味についても判断に迷うことはない．

Warning

このあたりで詰まったら

  • pushの実行時に何らかの認証ポップアップが表示される場合，第3節のSSH設定が未完了である可能性がある．ターミナルでssh -T git@github.comを再実行し，「successfully authenticated」と表示されるかを確認する．
  • Source Controlパネルに何も表示されない場合，開いているフォルダがgit-practiceそのものであるかを確認する（その内部にある別名のフォルダを開いてしまっていないか）．

6. GitHub Pagesによるページの公開

本節が最終的な到達点となる．memo.txtではなく，簡易なHTMLファイルを1件用意し，これを公開する．

6-1. 公開用HTMLファイルの作成

VS Codeにおいて，git-practiceフォルダ内にindex.htmlという名称の新規ファイルを作成し（左上の新規ファイルアイコンより作成できる），以下の内容をそのまま貼り付ける．

<!DOCTYPE html>
<html lang="ja">
<head>
  <meta charset="UTF-8">
  <title>はじめてのページ</title>
</head>
<body>
  <h1>こんにちは、世界！</h1>
  <p>これは私が初めてGitHubで公開したページです。</p>
</body>
</html>

ファイル名は必ずindex.htmlとする．これがWebページの「入口」として特別に扱われる規則上の名称であるためである．

6-2. 保存，およびadd → commit → push の実行

ファイルを保存（Cmd + S）した後，5-4で行った3手順（+ボタン→コミットメッセージ入力後✓→Sync Changes／雲のアイコン）を再度実行する．ターミナルで行う場合は，以下のコマンドによっても同様の結果が得られる．

git add index.html
git commit -m "公開用ページを追加"
git push

6-3. GitHub Pagesの有効化

ブラウザでGitHub上のリポジトリページを開き，以下の順序で進める．

  1. 「Settings」タブを選択する
  2. 左側メニューより「Pages」を選択する
  3. 「Build and deployment」の「Branch」において，mainと/(root)を選択し，「Save」を選択する

数十秒から1分程度の待機後，画面上部に公開用のURLが表示される（https://ユーザー名.github.io/git-practice/の形式）．

6-4. 完成の確認

表示されたURLをブラウザの新規タブで開く．「こんにちは、世界！」というページが表示されれば完成である．

これにより，ターミナル上で実行した数行のコマンドを起点として，世界中からアクセス可能なWebページが生成されたことになる．以降index.htmlを改変し，同一の手順（変更→add→commit→push）を繰り返すことで，このページは任意に更新可能である．

Warning

このあたりで詰まったら

  • Pagesの設定後もページが表示されない場合，反映までに数分を要することがある．時間を置いた上で再読み込みを行う．
  • 404ページ（見つかりません）が表示される場合，ファイル名が正確にindex.htmlであるか，6-3においてブランチがmainに設定されているかを再確認する．

7. 困った場合の3要点

全てを記憶する必要はないが，以下の3点は念頭に置くこととする．

  1. 現在位置が不明になった場合はpwd，フォルダ内部を確認したい場合はlsを実行する．
  2. 変更を記録する基本の流れは：ファイルの変更 → add（Definition 2への仮登録） → commit（Definition 3への記録） → push（GitHubへの送信）である（VS Codeの場合：ファイルの変更→＋ボタン→✓ボタン→雲のアイコン）．
  3. git statusは，Definition 1とDefinition 2の現状を示す安全装置である．不明な点があれば，まずこれを実行する．

8. 用語さくいん

これまでに登場した用語を，後から参照しやすいよう一覧にまとめる．記憶のためではなく，「これは何であったか」という確認の際に参照する辞書として用いることを想定している．

用語 内容
プロジェクト 管理対象とするファイル・フォルダ一式（Section 1.3を参照）
リポジトリ Definition 3を参照
ワーキングツリー Definition 1を参照
インデックス（ステージングエリア） Definition 2を参照
コミット Definition 4を参照
ローカルリポジトリ／リモートリポジトリ Definition 5を参照
SSHキー（秘密鍵／公開鍵） Definition 7を参照
git status Definition 1とDefinition 2の現状を表示するコマンド
git log これまでのコミット（履歴）を一覧表示するコマンド

なお，本稿では扱わなかったが，Gitの利用において頻出する用語が2件存在する．意味のみ簡潔に述べておく．

  • ブランチ：作業の続きを，本筋とは別に「仮に試みる」ための，もう一本の歴史の枝に相当するものである．本筋（main）に影響を与えずに試行が可能である．
  • マージ：枝分かれさせて試行した内容を，再び本筋（main）へ合流させる操作である．

本稿の範囲外とした事項

初回の作業において全てを扱うと混乱を招くため，意図的に範囲外とした事項が存在する．関心がある場合は，各自で調査するか，本稿を提供した者へ確認するとよい（Section 1.10に簡潔な意味は記載してある）．

  • ブランチ／マージ：分岐による試行・本筋への合流という操作自体は，本稿では扱わない
  • clone：第三者またはGitHub上のリポジトリを，自分のPCへ丸ごと取得する操作
  • pull／fetch：GitHub側の更新内容を，自分のPCへ取り込む操作（単独での作業中は出番が少ない）
  • コンフリクト：複数の作業者が同一箇所を同時に変更した場合に生じる衝突（単独での練習中には発生しない）
  • .gitignore：記録の対象から除外したいファイルを指定する設定

これらは，第三者と共同で同一のリポジトリを扱う機会が生じた際に，改めて必要となる事項である．現段階では，本日完了した内容のみで十分である．
